          О чем пойдет речь

⎼⎼⎼

Содержимое раздела касается проблем общего характера, связанных с возможностями которые дает правильное конфигурирование клиентского кэширования. А конкретно управление индексацией через правильную реакцию на заголовок if-modified-since.

Устранением таких проблем обычно занимается специалист с опытом системного администрирования.

         Глоссарий

⎼⎼⎼

Ниже описаны значения некоторых, используемых мной  терминов, смысл которых может отличаться от общепринятого.

Восприятие их именно такими, как описано ниже, важно для понимания вопросов, при описании которых я их использую.

        Ресурс

⎼⎼⎼

Любой массив данных, который можно получить зная его URL адрес. Например: HTML страницы, CSS, JS файлы, изображения, шрифты и т.д.

         Статика

⎼⎼⎼

Под термином статика подразумевается контент, который с высокой долей вероятности не будет претерпевать изменений (остается статичным)  в течении  всего времени своего существования или значимой его части.

Типичным примером подобных ресурсов являются изображения, файлы CSS стилей, часть JavaScript файлов, шрифты и так далее.

         Динамика

⎼⎼⎼

Это ресурсы, которые своими качествами противоположны статике.
Чаще всего это страницы проекта, контент которых дополняется, изменяется, совершенствуется.

         Клиент

⎼⎼⎼

Любой агент инициирующий процедуру загрузки  ресурса.  
Например браузер

         Пользователь

⎼⎼⎼

Любой агент способный  инициировать  работу клиента.
Например человек, или алгоритм действующий согласно заложенной в него логике.

         Кэш

⎼⎼⎼

Комплекс технических средств, задачей которых является, обеспечить максимально быстрый доступ к ранее полученным данным.

         Поисковая система

⎼⎼⎼

Набор технических средств, решающих вопросы агрегирования контента online проектов, с последующим его дифференцированием, согласно заложенной в них моделью  релевантности.

Например Google, Яндекс и т.д.

Время жизни кэша

⎼⎼⎼

         Описание проблемы

⎼⎼⎼

        Минимальное  время жизни кеша ⓘ  - должно быть не менее 1 года.

Сервера проекта отдают статический контент без явно установленных заголовков Cache Control или Expires, явно задающих время жизни кэша:

$ curl -IL https://example.com/image.png

HTTP/1.1 200 OK
Server:            nginx/1.14.0
Content-Type:      image/png
Content-Length:    1356
Last-Modified:     Fri, 25 Jan 2019 14:52:48 GMT

В таких случаях клиент начинает опираться на значение заголовка last-modified и рассчитывает время по специальной формуле.

         Формуле детально описана здесь ⓘ .

         Решение

⎼⎼⎼

Необходимо увеличить время жизни кеша до 1 года.

ETag

⎼⎼⎼

         Описание проблемы

⎼⎼⎼

В каждом ответе сервера сейчас присутствует заголовок ETag

Если у проекта нет никаких механизмов использования значения ETag, то ресурсы расходуемые на формирование этого заголовка тратятся впустую.

$ curl -IL https://example.com/image.png

HTTP/1.1 200 OK
ETag:              5c4b22c0-54c

Иными словами, если в вашем проекте нет какой то специальной логики привязанной к ETag это означает что сервер тратит достаточно много ресурсов на его формирование и при этом делает это зря, так как фактически просто дублирует функционал заголовка cache-control с значением max-age

If-Modified-Since

⎼⎼⎼

         Описание проблемы

⎼⎼⎼

        Корректная обработка запросов, с установленным заголовком  If-Modified-Since это ключевой механизм эффективного  распределения крауленгового  бюджета.

У проекта верно сконфигурирована логика формирования таких ответов для статического контента, однако, в силу ошибки конфигурирования веб сервера Nginx , отвечающего за раздачу статики, поисковой робот не получает верного ответа.

         Решение

⎼⎼⎼

Причиной проблемы служит значение по умолчанию в конфигурации Nginx

if_modified_since exact;

Которое нужно изменить на before

if_modified_since before;

If-Modified-Since

⎼⎼⎼

        Описание задачи

⎼⎼⎼

На текущий момент, кэширование  для динамики отключено полностью, то есть ни одного заголовка управляющего клиентским кэшем не установлено:

$ curl -IL https://example.com/Cialis-for-Women.html

HTTP/1.1 200 OK
Server:            nginx/1.14.0
Content-Type:      text/html; charset=UTF-8
X-Powered-By:      PHP/5.4.16
X-Frame-Options:   SAMEORIGIN

С динамикой, настоятельно  рекомендуется работать точно так же как и с статикой.  То есть реализовать полноценное кеширование.  И сделать это более чем по силам, в силу того, что контент не меняется каждую секунду.

        Почему это важно для SEO  ⓘ 

         Решение

⎼⎼⎼

Если дата последней модификации контента страницы известна, то в обязательном порядке следует реализовать правильную полноценную работу  с ответом на запросы с заголовком If-Modified-Since.

Если условия проекта, требуют, чтобы посетитель видел гарантированно актуальную версию страницы, то есть мы не можем себе позволить, чтобы при запросе страницы посетитель увидел кэшированную версию даже на короткий период, можно применить прием с принудительной валидацией.

         Принудительная валидация кеша  ⓘ 

Если по политическим причинам (а технических причин препятствующих этому нет), реализация клиентского кэша для динамики невозможна в принципе, можно попробовать пойти на компромисс:

Работа для пользователя сохраняется в текущем режиме. Работа для поискового робота изменяется. К проекту дописывается небольшой модуль, который:

  1.  Для каждой страницы, которая должна быть в индексе, отслеживает и сохраняет дату захода бота на эту страницу, при условии что бот зашел с установленным заголовком If-Modified-Since
  2. Модуль должен уметь отвечать на вопрос, есть ли страницы где бота не было N дней от текущего момента.
  3. В Модуле реализовывается логика, которая, при следующем посещении бота страницы,  с обозначенным заголовком проверяет:
  1. Запрошенная страница имела посещение бота ближайшие N дней? Если ответ положительный, то возвращает боту ответ 304 и обрывает соединение. Если отрицательный то следующий шаг.
  2. Существуют ли страницы которые не посещались ботом ближайшие N дней. И если ответ положительный, то возвращает  ответ 304 и обрывает соединение.  Если отрицательный то следующий шаг.
  3. Выдает текущую страницу, и отмечает у себя дату.  

Цифра N это искусственным образом выбранная цифра, определяющая период  когда мы считаем что контент на странице не изменяется. Например 2 будет означать что мы считаем, что в течении двух дней с момента индексации контент неизменный.

Подбирается цифра из расчета, чтобы все страницы проекта индексировались не реже реального обновления контента.  
Устанавливается экспериментальным путем.
Сначала ставим 14 дней, отслеживаем состояние,
видим что взяли с запасом - уменьшаем.

Таким образом мы искусственно создаем условия, при которых не бот выбирает что ему индексировать, а мы ему определенно это сообщаем.  Как итог идеальная скорость переиндексации  и возможность при необходимости засунуть в индекс любой новый контент в течении 5 минут после его публикации.

Клиентское кэширование

⎼⎼⎼

        Описание

⎼⎼⎼

        Клиентское кэширование - это возможность объяснить клиенту ⓘ, какой ресурс и при каких условиях, может использоваться повторно без необходимости в его повторной загрузке.

Длинный ответ:
Клиентское кэширование, это функционал, реализованный в рамках протокола HTTP,  который позволяет браузеру, в процессе загрузки ресурса, получить от сервера описание условий, соблюдая которые, загруженный ресурс, может быть сохранен в локальном кэше, с целью его последующего использования, минуя фазу повторной загрузки.

        RFC 7234 

Поисковые системы ⓘ используют  протокол клиентского кэширования, как источник актуальной информации о текущих изменениях проекта.

Настроенный клиентский кэш ⓘ , посредством минимизации затрачиваемых клиентом ресурсов, оказывает прямое влияние на качество взаимодействия клиента с проектом. Что оказывает косвенное влияние на положение проекта в поисковой выдаче.

        Чем это полезно для SEO

⎼⎼⎼

Так как  страницы, потенциально, могут регулярно претерпевать изменения, а в интересах поисковой системы предоставлять посетителю актуальную информацию, поиск вынужден разрешать задачу не просто индексации ресурсов проекта, но задачи постоянной переиндексации страниц, которые поиск обозначил для себя как важные.

        Работа поискового алгоритма с проектом,  представляет из себя непрерывный процесс постоянных переиндексаций всех его ресурсов.

Частота переиндексирования индивидуальна как для любого проекта в целом, так и для каждого его ресурса в частности, и зависит от внутренних процессов поисковой машины.

Поисковый робот, также как и браузер посетителя,  на основании заголовков управляющих клиентским кэшированием, может принимать решение о том, следует ли повторно загружать ресурс, или можно воспользоваться данными из предыдущей сессии работы с проектом.

То есть, при правильной настройке клиентского кэширования, поисковый робот, будет получать исчерпывающую информацию о том, какие ресурсы проекта изменились и, как следствие, требуют повторной индексации, а какие остались прежними, в следствии чего не требуют сейчас  обработки.

Организация именно такой логики работы создает все условия для оперативной индексации любой изменившейся страницы проекта.

        Правильно настроенное клиентское кэширование это один из самых эффективных способов управления процессом индексации проекта поисковыми системами.

        Принцип работы

⎼⎼⎼

Серверная часть, обслуживая запрос, добавляет к ответу ряд служебных заголовков,  содержимое которых, дает возможность клиенту определить для себя допустимые методы работы с этим ресурсом, для случаев, когда этот ресурс будет запрошен повторно.
Такое поведение называют  выбором стратегии кэширования. 

Основные варианты:

  1. Обозначить для клиента время,  в течении которого он может использовать ресурс без необходимости скачивать его повторно. То есть использовать локальную копию.

Такой промежуток времени называют временем жизни кэша.

  1. Обозначить клиенту способ, используя который он может удостовериться в актуальности локальной копии ресурса.

Такой процесс называют валидацией кэша.

  1. Сообщить клиенту о том, что доступ к ресурсу может осуществляться исключительно через новый сетевой запрос к серверу. Создавать и использовать любые локальные копии ресурса запрещено.

Исторически сложилось, что любую ситуацию, когда клиент вынужден совершить сетевой запрос для получения ресурса,  называют ситуацией, когда кэширование отключено, что не является правильным пониманием процесса.

Разница между запретом создавать локальные копии, и как следствие обязательным сетевым запросом, и ситуацией когда такой же запрос необходимо сделать в силу созданных обстоятельств, (например установленном времени жизни кэша в отрицательное значение) заключается в том, существует ли локальная копия в кэше браузера. В первом случае нет. Во втором случае да. И она может использоваться.

Наглядно это демонстрирует ситуация, когда в процессе работы, пропадает возможность осуществлять сетевые запросы (пропал интернет).

В случае запрета создавать локальные копии, клиент может вернуть пользователю только сообщение о том, что связи нет.

В случае же отрицательного времени жизни кэша, браузер может вернуть текущую просроченную локальную копию, с сообщением объясняющим происходящее.

        Ситуация, когда при запросе ресурса, клиент совершает сетевой запрос, не говорит о том, что ресурс отсутствует в кэше, или кэш отключен.  

Но говорит только о том, что в текущих условиях, браузер должен обратиться к серверу, вне зависимости от того есть локальная копия ресурса или ее нет.

Потому сталкиваясь с инструкциями, где для отключения кэширования рекомендуют выставить, например, время жизни кэша в отрицательное значение, мы на самом деле кэш не отключаем, а только ограничиваем множество ситуаций, в которых клиент может использовать локальную копию.

        Схема выполнения запроса клиентом

⎼⎼⎼

Схема носит упрощенный характер, но покрывает собой большую часть всего множества случаев обработки запросов.

        ⓘ info:
Время жизни кэша

⎼⎼⎼

Чтобы определить срок, в течении которого, запрошенный ресурс будет сохранять актуальное состояние, а значит не будет необходимости в его повторном скачивании, используется один из трех http заголовков: cache-control, expires или last-modified.

         Заголовок cache-control

⎼⎼⎼

Заголовок cache-control является частью стандарта HTTP/1.1 и рекомендован к использованию как промышленный стандарт в вопросах клиентского кэширования.

Это заголовок, который может содержать большое количество данных, связанных с вопросами кэширования. Данные перечисляются через запятую в формате: параметр = значение, параметр = значение и т.д.

В рамках вопроса об управлении временем жизни кэша, используется параметр с именем  max-age.

Значение параметра представляет собой количество секунд,  добавив которые к дате запроса, мы получим дату, до наступления которой, локальная копия ресурса сохраняет свою актуальность.

Например:
чтобы определить для локальной копии время жизни в один год, сервер установит значение параметра в
365 дней * 24 часа * 60 минут * 60 секунд = 31 536 000

cache-control: max-age=31536000

В течении этого времени, клиент не будет осуществлять повторных обращений к серверу с целью получения этого ресурса, а будет использовать его локальную копию. Если, конечно, она есть.

        Заголовок expires

⎼⎼⎼

Заголовок expires это часть стандарта HTTP/1.0. 

        Так как стандарт HTTP/1.0 является устаревшим, использование заголовка expires не рекомендуется.

Тем не менее, заголовок поддерживается практически любой серверной и клиентской платформой.

Содержимое заголовка представляет из себя дату до наступления которой ресурс сохраняет свою актуальность.

Дата задается в формате описанном в  RFC 7231  и имеет вид:
<имя-дня>, <номер-дня> <имя-месяца> <год> <час>:<минута>:<секунда> GMT

Например, чтобы задать дату равную 21 апреля 2020 года 5:45 утра

expires: Thu, 21 Apr 2020 05:45:39 GMT

        Если одновременно будут установлены и  cache-control max-age и expires, значение из expires будет проигнорировано

                 Заголовок last-modified

____

Значение заголовка last-modified устанавливается равным  дате последней модификации ресурса.

Формат даты аналогичен формату заголовка expires
<имя-дня>, <номер-дня> <имя-месяца> <год> <час>:<минута>:<секунда> GMT

last-modified: Wed, 03 Apr 2019 10:16:16 GMT

В том случае, если заголовки cache-control и expires не установлены, но при этом установлен  заголовок last-modified, то время жизни кэша рассчитывается по формуле:

( ( текущая дата — значение last-modified ) / 10 ) + текущая дата

        Основное назначение заголовка last-modified это разрешение вопросов актуальности контента ресурса, о чем пойдет речь ниже.

        Если заголовки не установлены

⎼⎼⎼

Если ни один из заголовков не установлен, то время жизни кэша принимается равным нулю.

В этом случае, клиент, при повторном запросе,  автоматически считает кэш просроченным и сразу переходит к процедуре валидации кеша.

        ⓘ info:
Валидация кэша

⎼⎼⎼

Если во время запроса, который привел к созданию локальной копии ресурса, серверная часть определила значения заголовка last-modified и/или заголовка ETag, то клиент, при следующем запросе, при наличии локальной копии ресурса,  обязательно добавит данные этих заголовков к запросу.

Серверная часть, обслуживая запрос с такими заголовками, проведет в первую очередь процедуру валидации данных из заголовков. В случае успеха, вернет клиенту только короткий ответ означающий, что локальная копия может продолжать использоваться дальше.  

Получив ответ, клиент обновляет заголовки кеширования в соответствии с текущим ответом, и возвращает локальную копию пользователю.

        Last-modified

⎼⎼⎼

Содержимое  заголовка last-modified,  интерпретируется как дата последней модификации запрошенного ресурса.

last-modified: Wed, 03 Apr 2019 10:16:16 GMT            

Клиент при отправке запроса серверу, в ситуации когда:

  1. существует локальная копия ресурса
  2. при ее создании, в запросе был установлен заголовок last-modified

к запросу добавляет заголовок If-Modified-Since с значением которые было в заголовке last-modified

        If-Modified-Since

⎼⎼⎼

Значение заголовка может быть абсолютно любой датой. Обычно, устанавливают равным значению  из найденного  ранее last-modified:

If-Modified-Since: Wed, 03 Apr 2019 10:16:16 GMT

Сервер, получив запрос с таким заголовком,  извлекает из него дату и сравнивает ее с текущей датой модификации запрашиваемого ресурса  (например Wed, 10 Apr 2019 10:16:16 GMT).

Если дата модификации известная серверу, ближе к текущей дате, чем дата из  присланного клиентом заголовка,  то сервер отвечает http кодом 200 и инициирует процедуру передачи запрошенного ресурса так, как это было при первом запросе.

Если дата, известная серверу, идентична или дальше от текущей даты, чем дата из присланного заголовка, то, сервер возвращает http код 304, добавляет к ответу актуальные значения для заголовков,  и обрывает соединение.

Клиент получив код ответа 304, возвращает пользователю локальную копию и обновляет для нее заголовки кеширования в соответствии с полученным ответом.

        Корректная обработка запросов, с установленным заголовком  If-Modified-Since,  является ключевым способом по управлению того, что индексирует поисковая система.

Способом, который позволяет предотвратить повторную индексацию ресурсов, которым эта индексация не нужна.

        Google, Яндекс и Last-modified

⎼⎼⎼

        И Google и Yandex,  запрашивая ресурс, который уже был проиндексированный ими ранее, в обязательном порядке  устанавливают заголовок  If-Modified-Since. 

Значение заголовка устанавливается равным дате предыдущего сканирования этого ресурса.

В результате чего сервер может  реагировать 304 ответом в случае если  ресурс не изменился, на что поисковая система реагирует переходом к следующему ресурсу.  

Результатом этого становится эффективное распределение внимания поисковой системы,  к индексируемым ресурсам.

        Etag

⎼⎼⎼

Это заголовок, который выполняет по сути ровно те же задачи, что и  last-modified.  То есть предоставление механизма для валидации локальной копии ресурса.

Ключевое отличие в том, что значение, которое используется в заголовке, не является датой, а является некоторым уникальным идентификатором:

etag: W/"5cd958ff-7096"

Клиент при отправке запроса серверу, в ситуации когда:

  1. существует локальная копия ресурса
  2. при ее создании, в запросе был установлен заголовок etag

к запросу добавляет заголовок If-None-Match с значением которые было в заголовке etag

        if-None-Match

⎼⎼⎼

Значение заголовка делается равным etag заголовку установленному в момент формирования локальной копии

If-None-Match: W/"5cd958ff-7096"

Сервер, получив запрос с заголовком If-None-Match, проводит вычисление хеш функции для запрашиваемого ресурса, результат которого сравнивается с значением из заголовка.

Если значения отличаются, сервер отвечает http кодом 200 и запускает процесс ответа на запрос идентичный тому, как если бы ресурс запрашивался впервые.

Если же результат хеш функции совпал с данными из заголовка, то сервер возвращает http код 304, добавляет к ответу актуальную информацию по заголовкам и завершает соединение.  

Клиент, получив 304 код ответа, возвращает пользователю локальную копию ресурса.

        Алгоритм формирования Etag

⎼⎼⎼

Алгоритм хэш-функции может быть совершенно любым. НЕ существует спецификации которая бы описывала это процесс. По большому счету, это не требуется, так как принцип валидации ресурса, заключается в работе с некоторым магическим идентификатором, принцип формирования которого, владельцу проекта известен.

В Web сервере Apache, по умолчанию, используется алгоритм, отдаленно напоминающий CRC32 - контрольную сумму файла. Из чего следует вывод:

        Количество ресурсов, требуемых для расчета значения по умолчанию, для заголовка Etag, многократно превышает ресурсы, требуемые для получения даты модификации.

Необходимость в таком механизме работы с кэшем, возникает в тех случаях, когда архитектура проекта не позволяет отследить дату модификации  ресурса.  

        Благодаря возможности задать свою логику формирования магического значения etag,  механизм на основе etag и If-None-Match, часто используется для разрешения специфических задач проекта.

Например трекер пользователя, которым, в отличии от трекеров на основе Cookie намного сложнее манипулировать на стороне клиента, поскольку только крайне специфичные клиенты предоставляют интерфейс к управлению сохраненными заголовками кэширования.

Не говоря уже о том, что подобным образом "трекаются" промежуточные прокси сервера.

        Использование механизмов на основе etag, в странах евросоюза,  предполагает обязательное раскрытие алгоритма формирования заголовка.

А в случае использования каких либо данных посетителя, обязывает к информированию последнего о происходящем. 

        Google, Яндекс и Etag

⎼⎼⎼

Google и Яндекс не используют в своей работе заголовок etag.  

Использование заголовка etag не дает никаких преференций в плоскости SEO продвижения.

        Google и Яндекс не используют в своей работе заголовок ETag. 

Использование заголовка ETag не дает никаких преференций в плоскости SEO продвижения.

        Как отключить ETag

⎼⎼⎼

Так как, практически все задачи кэширования разрешаются за счет использования  Last-modified / If-Modified-Since, а сам Etag не предоставляет иного функционала то использование его избыточно.  

С учетом того,  что в случаях работы со статикой большого объема, формирование Etag может приводить к неоправданно высокому использованию вычислительных ресурсов сервера, рекомендуется:

        Если у проекта нет своих собственных внутренних специфических задач, требующих формирования etag заголовка, рекомендуется, полностью отключать его  формирование.  

В Apache:

FileETag None;

В Nginx:

etag off;

        Etag и Last-modified

⎼⎼⎼

В ситуации когда в запросе, на основе которого сформирована локальная копия ресурса, были установлены заголовки  и Etag и Last-modified,  клиент, при формировании следующего запроса отправит значение обоих заголовков.

Сервер должен проверить на валидность значение обоих заголовков и вернуть ответ 304 только в случае, если оба значения корректны.

Если один из механизмов сигнализирует о несоответствии копии оригиналу, сервер должен вернуть код 200 и актуальную версию ресурса.

        ⓘ info:
Принудительная валидация

⎼⎼⎼

Принудительная ревалидация это устоявшийся(разговорный) термин, обозначающий ситуацию когда клиент, при каждом запросе, несмотря на наличие локальной копии, отправляет запрос серверной части, с заголовками валидации.

Это может быть полезно, например, в ситуации, когда мы знаем что ресурс измениться, но как скоро сказать не можем. При этом стоит задача максимально оперативной доставки контента посетителю.

Добиваются этого путем

  1. установки времени кэширования равным нулю
  2. удалением любых заголовков связанных с ременем кэширования, но сохранением заголовков валидации кэша
  3. установкой специфического параметра   no-cache у заголовка cache-control

        Параметр no-cache не препятствует клиенту в создании локальных копий ресурса, но говорит только о том, что прежде чем использовать локальную копию, нужно отправить запрос серверной части.

Например установив:

cache-control: no-cache
last-modified: Wed, 03 Apr 2019 10:16:16 GMT

мы заставляем клиент создать локальную копию ресурса, и в случае повторного запроса - отправить  запрос к серверу, используя заголовок If-Modified-Since со значением из last-modified.

        ⓘ info:
Несколько копий ресурса

⎼⎼⎼

Идентификация ресурса на стороне клиента происходит не только по его URL.

Строго говоря для идентификации локальной копии,  используется ее  URL плюс дополнительные условия, которые задаются в HTTP заголовке Vary.

Если заголовок Vary не установлен, то идентификация ресурса происходит только на основе запрашиваемого URL.

Если заголовок Vary задан, то он должен содержать список HTTP заголовков, которые нужно использовать дополнительно для идентификации ресурса.

Например при запросе ресурса https://example.com/image.jpg  заголовок:

vary: user-agent

сообщает клиенту о том, что полученный ресурс можно отдавать из кэша только в случае, если user-agent совпадает с тем, который был на момент первого запроса.

Таким образом мы можеме привязывать конркетные версии ресурсов к специфической  версии клиента (заголовок Referer), его функциональности (заголовок  Accept) или чего-то экзотического используя заголовок Cookie.

        ⓘ info:
Локальные копии

⎼⎼⎼

        Запрет/разрешение на создание

⎼⎼⎼

Информирование о том, запрещено или разрешено создавать локальные копии, происходит посредством  установки параметра  no-store в http заголовке cache-control 

cache-control: no-store

Получив подобный заголовок, клиент обязан:

  1. в процессе своей работы хранить копию не дольше чем нужно для ее текущей визуализации. Это касается любых структур данных, обеспечивающих доставку ресурса: значение заголовков, дат и т.д.
  2. препятствовать сохранению копии на долговременных носителях;
  3. при повторном запросе, формировать условия, которые характерны для первой загрузки ресурса.
  4. процесс не может быть регламентирован пользовательскими настройками. То есть пользователь не должен иметь возможности вмешиваться в ход этого процесса.

        ⓘ info:
Промежуточные сервера

⎼⎼⎼

Нередки случаи, когда между клиентом и сервером, присутствуют промежуточные кэширующие сервера.

Устанавливая в заголовке  cache-control, параметры public и private, сервер дает понять всем кэширующим серверам, через которые проходит запрос, каким образом они могут использовать этот ресурс.

Параметр  public сообщает о том, что кэш можно возвращать любому другому пользователю на тот же самый запрос.

Параметр private разрешает возвращать текущий кэш только тому же самому пользователю.

Подобное поведение крайне полезно в ситуациях работы с индивидуальным кэшем для, например, авторизованных пользователей.

Проверяем If-Modified-Since

⎼⎼⎼

Как писалось выше ⓘ , обеспечение корректной обработки If-Modified-Since имеет чрезвычайно важную роль в случае, если мы боремся за органический трафик.

Ниже перечислены способы диагностирования корректной обработки сервером этого заголовка.

        Online

⎼⎼⎼

Для проверки можно воспользоваться online сервисом 

        Curl

⎼⎼⎼

передаем текущую дату в формате  dd mmm YYYY :

$ curl -I -z '13 Jan 2020' https://example.com/img/image.jpg

Ответ должен быть 304

HTTP/1.1 304 Not Modified
Expires: Tue, 02 Mar 2032 13:40:46 GMT
Cache-Control: max-age=383616000, public

        JavaScript / Node / Google Developer Tools

⎼⎼⎼

Открыв любой агент который поддерживает выполнение JS кода, например Developer Tools в браузере Google Chrome, достаточно в контексте ресурса выполнить команду

fetch("https://examples.com", {
headers: {
"if-modified-since": new Date().toGMTString()
}
})
.then(function(r) {
r.status === 304 ? console.log(
 "%cAll fine. Response status code: %d. \nResponse object: %O",
 "color: green",
  r.status,
  r)
 : console.log("%cResponse status code: %d when expected: 304.
  \nResponse object: %O",
   "color:red",
   r.status,
   r);
})
.catch(function(r) {
   console.log("%cSomthing wrong\n" + r, "color:red");
});

Примеры конфигураций

⎼⎼⎼

Ниже примеры того, каким образом можно управлять временем жизни кэша в случаях использования распространенных WEB серверов, языков программирования или популярных фреймворков.

        Apache

⎼⎼⎼

Для типичного набора статических ресурсов (изображения, CSS и JS файлы и т.д.)  устанавливаем время жизни кеша в 1 год,

<filesMatch "(?i).(ico|pdf|flv|jpg|jpeg|png|gif|js|css|webp)$">
   Header set Cache-Control "max-age=383616000"
</filesMatch>

Незабываем удостовериться в том, что модуль headers подключен.

В случае если необходим expires, для файла с типом jpg

ExpiresActive On
ExpiresByType image/jpeg "now plus 148 months"

Незабываем удостовериться в том, что модуль expires подключен и в конфигурации Apache определен тип файла заданный в правиле
expires (image/jpeg):

0        beshort                0xffd8                image/jpeg

⎼⎼⎼

location ~* ^.+\.(jpg | gif | png | css | js | woff | webp )$ {
expires max;
}

server.modules  = (
"mod_expire",
"mod_setenv",

setenv.add-response-header += ("Cache-Control" => "public")
)

        Популярные фреймворки

⎼⎼⎼

Если разработка ведется с использованием какого-либо фреймворка (Laravel, Yii, Ruby и т.д.) то решать вопросы заголовков для формируемых страниц нужно используя предусмотренные фреймворком интерфейсы.

Например для Yii2 код выглядеть может следующим образом:

public function behaviors() {
return [
   [
       'class' => 'yii\filters\HttpCache',
       'only' => ['index'],
       'lastModified' => function ($action, $params) {
           $q = new \yii\db\Query();
           return $q->from('post')->max('updated_at');
       },
   ],
];
}

Nginx и If-Modified-Since

⎼⎼⎼

Конфигурация Nginx, по умолчанию, обслуживает запрос с if-modified-since только в случае указания даты абсолютно идентичной той, которая отдается при запросе. Что естественно приводит к ситуации, когда поисковые машины, которые указывают дату последнего сканирования ресурса, получают вместо ожидаемого кода 304,  код  200. То есть инициируют процедуру переиндексации.

Причиной этому служит значение по умолчанию для параметра

if_modified_since exact;

Нужно в файле конфигурации Nginx изменить значение exact на

if_modified_since before;

Документация Nginx о if_modified_since 

        Письмо хостеру

⎼⎼⎼

        В случае если доступа к конфигурационным файлам Nginx хостер не предоставляет, ему можно отправить следующее письмо:

Здравствуйте.

Текущая конфигурация Nginx, при запросе статического контента,   (.jpg .png .css и т.д.) с установленным заголовком if_modified_since корректно работает только в случае точного совпадения даты из этого заголовка с датой из заголовка last-modified.

Происходит это потому, что по умолчанию Nginx работает с настройкой

if_modified_since exact;

Очень прошу Вас исправить это, то есть прописать явно в конфигурации Nginx настройку:

if_modified_since before;
